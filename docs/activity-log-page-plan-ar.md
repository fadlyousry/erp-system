# فكرة صفحة سجل العمليات

## الهدف
إنشاء صفحة داخل النظام تعرض "سجل العمليات" بحيث أي إجراء مهم يقوم به المستخدم يتم تسجيله والرجوع له لاحقًا للمراجعة، المتابعة، ومعرفة من قام بأي تعديل وفي أي وقت.

## لماذا الصفحة مهمة؟
- متابعة نشاط المستخدمين داخل النظام.
- مراجعة الأخطاء أو التعديلات غير المتوقعة.
- معرفة من أنشأ أو عدل أو حذف البيانات.
- تحسين الرقابة الداخلية والمحاسبية.
- تسهيل التحقيق في المشاكل عند حدوث فرق في الأرصدة أو الفواتير أو المخزون.

## ما الذي يتم تسجيله؟
يفضل في المرحلة الأولى تسجيل العمليات المهمة فقط، مثل:
- تسجيل الدخول وتسجيل الخروج.
- إنشاء مستخدم أو تعديل صلاحياته.
- إنشاء فاتورة بيع أو تعديلها أو حذفها.
- إنشاء مرتجع بيع أو تعديل بياناته.
- إنشاء فاتورة شراء أو مرتجع شراء.
- إضافة دفعة عميل أو مورد أو تعديلها أو حذفها.
- إضافة مصروف أو تعديله أو حذفه.
- أي حركة على الخزنة أو تحويل بين الخزن.
- إضافة منتج أو تعديل سعره أو وحدته أو مخزنه.
- تعديل الإعدادات العامة للنظام.

## البيانات التي يجب حفظها لكل عملية
كل سجل عملية يفضل أن يحتوي على:
- `id` رقم السجل.
- `userId` رقم المستخدم.
- `userName` اسم المستخدم وقت التنفيذ.
- `action` اسم العملية نفسها.
- `module` اسم القسم مثل المبيعات أو المنتجات أو الخزنة.
- `entityType` نوع العنصر المتأثر مثل `sale` أو `product` أو `expense`.
- `entityId` رقم العنصر المتأثر.
- `description` وصف مختصر مفهوم للبشر.
- `oldValues` القيم القديمة قبل التعديل.
- `newValues` القيم الجديدة بعد التعديل.
- `status` هل العملية نجحت أم فشلت.
- `ipAddress` أو `deviceInfo` لو متاح.
- `createdAt` وقت تنفيذ العملية.

## شكل الصفحة المقترح
صفحة سجل العمليات تكون بسيطة وواضحة، وتشمل:
- شريط فلاتر بالأعلى.
- جدول رئيسي لعرض السجلات.
- نافذة تفاصيل لكل سجل.

### الفلاتر المقترحة
- فلتر بالتاريخ: من / إلى.
- فلتر بالمستخدم.
- فلتر بالقسم.
- فلتر بنوع العملية.
- فلتر بالحالة: نجاح / فشل.
- بحث نصي سريع.

### أعمدة الجدول المقترحة
- التاريخ والوقت.
- المستخدم.
- القسم.
- العملية.
- العنصر المتأثر.
- الوصف.
- الحالة.
- زر عرض التفاصيل.

### تفاصيل السجل
عند فتح السجل تظهر:
- الوصف الكامل.
- القيم القديمة والجديدة.
- رقم المرجع المرتبط.
- اسم الشاشة أو الموديول.
- وقت العملية بدقة.

## ملاحظات مهمة في التنفيذ
- لا نسجل كل شيء من البداية حتى لا يكبر حجم البيانات بدون داعٍ.
- نبدأ بالعمليات الحساسة ماليًا وإداريًا.
- يجب أن يكون التسجيل في الباك إند وليس الواجهة فقط.
- يفضل أن تكون كتابة السجل جزءًا موحدًا من الخدمات بدل تكرار الكود في كل مكان.
- يفضل إخفاء البيانات الحساسة مثل كلمات المرور.
- يجب تحديد صلاحية خاصة لعرض سجل العمليات.

## تصور تقني مبدئي
الهيكل المتوقع في المشروع:
- جدول جديد في قاعدة البيانات مثل `ActivityLog`.
- دوال في `electron/db-service.js` أو خدمة مستقلة خاصة بالسجل.
- IPC handlers لجلب السجلات من قاعدة البيانات.
- API في `preload.js`.
- صفحة جديدة في `src/pages` مثل `ActivityLog.jsx`.

## التاسكات المطلوبة

### المرحلة 1: التحليل والتصميم
- تحديد قائمة العمليات التي يجب تسجيلها في أول إصدار.
- تحديد أسماء الأقسام والـ actions بشكل موحد.
- تحديد البيانات الحساسة التي لا يجب تخزينها.
- تحديد من يملك صلاحية مشاهدة الصفحة.

### المرحلة 2: قاعدة البيانات
- إنشاء موديل `ActivityLog` في `prisma/schema.prisma`.
- تحديد الحقول المطلوبة وأنواعها.
- إضافة الفهارس المناسبة على `userId`, `module`, `createdAt`.
- إنشاء migration جديدة.

### المرحلة 3: منطق التسجيل
- إنشاء helper موحد مثل `logActivity`.
- ربط الـ helper بالعمليات الأساسية المهمة.
- تسجيل الإنشاء والتعديل والحذف بشكل واضح.
- إضافة دعم لتسجيل النجاح والفشل.

### المرحلة 4: الربط مع Electron
- إضافة IPC handler لجلب السجلات.
- إضافة IPC handler لجلب تفاصيل سجل واحد عند الحاجة.
- إضافة فلاتر واستعلامات pagination للأداء.
- كشف الدوال المطلوبة في `preload.js`.

### المرحلة 5: الواجهة
- إنشاء صفحة `ActivityLog.jsx`.
- بناء شريط الفلاتر.
- بناء جدول عرض السجلات.
- بناء مودال أو Drawer لتفاصيل السجل.
- ربط الصفحة بالـ routing والقائمة الجانبية.

### المرحلة 6: الاختبارات والتحسين
- اختبار تسجيل العمليات من أكثر من شاشة.
- التأكد أن الأداء جيد مع عدد كبير من السجلات.
- اختبار الفلاتر والبحث والترتيب.
- التأكد أن السجل لا يكشف بيانات حساسة.
- مراجعة صلاحيات الوصول للصفحة.

## أولويات التنفيذ المقترحة
لو الهدف تنفيذ سريع وعملي، نبدأ بالآتي:
1. تسجيل عمليات المبيعات.
2. تسجيل عمليات الخزنة.
3. تسجيل الدفعات.
4. تسجيل تعديل المنتجات.
5. تسجيل إعدادات النظام والمستخدمين.

## نتيجة الإصدار الأول المقترحة
أول نسخة من الصفحة تكون كافية إذا كانت:
- تعرض السجلات بفلترة جيدة.
- تسجل أهم العمليات الحساسة.
- توضح من قام بالعملية ومتى.
- تعرض القيم قبل وبعد التعديل في العمليات المهمة.
- تعمل بسرعة بدون التأثير على الاستخدام اليومي.
